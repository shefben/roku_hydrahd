# HydraHD brs-engine debug harness

Runs the HydraHD Roku channel in a real browser using the
[brs-engine](https://github.com/lvcabral/brs-engine) BrightScript/SceneGraph
simulator, so the UI can be navigated and screenshotted for visual / layout /
navigation debugging **without a physical Roku**.

It is driven by the `/roku-visual-debug` skill (see
`.claude/skills/roku-visual-debug/`), but you can also use it by hand.

## What's here

| File | Purpose |
|------|---------|
| `serve.js` | Dependency-free static server. Sets the COOP/COEP headers that `SharedArrayBuffer` needs (cross-origin isolation) and hosts a `/proxy/` CORS proxy so the channel can fetch real hydrahd data. |
| `index.html` | Page with the 1920×1080 `#display` canvas + hidden `#player` video. |
| `app.js` | Boots the engine and exposes automation hooks on `window` (see below). |
| `rebuild.ps1` | Rebuilds the channel `.zip` and stages it into `channel/`. |
| `lib/` | `brs.api.js`, `brs.worker.js` (engine 2.2.0) + `brs-sg.js` (scenegraph 0.2.0). |
| `assets/common.zip` | SceneGraph fonts/resources volume (from `brs-scenegraph`). |
| `channel/HydraHD.zip` | The staged channel build (regenerated by `rebuild.ps1`). |
| `screenshots/` | Output screenshots. |

## Quick start

```powershell
# one-time
cd tools/brs-debug
npm install

# each session
./rebuild.ps1            # build + stage the channel zip
node serve.js            # serve at http://127.0.0.1:6502/  (leave running)
```

Then open `http://127.0.0.1:6502/?status=1` in Chrome.

- `?status=1` shows the engine status line (bottom-left).
- `?proxy=0` disables the CORS proxy (channel runs offline; rows show "Failed to load").
- `?zip=channel/other.zip` loads a different package.

After editing channel source, run `./rebuild.ps1` and refresh the tab.

## Automation hooks (on `window`)

| Hook | Description |
|------|-------------|
| `brsState()` | `{ status, ready, lastError, eventCount, consoleTail }`. `status` reaches `"started"` when the channel is running. |
| `brsKey(name, delayMs?)` | Send one remote key. Names: `up down left right · select/ok/enter · back/esc · home · info/options/star · play/pause · rev fwd · replay`. |
| `brsKeys(seq, gapMs?)` | Send a sequence, e.g. `brsKeys("down down select")`. Returns a Promise. |
| `brsShot()` | Full-resolution PNG data URL of the current frame (independent of CSS scale). |
| `brsErrors()` | Captured error/crash lines (main thread). |
| `__brsEvents` | Array of every engine lifecycle event `{id,data,t}`. |

BrightScript `print` output and runtime errors run on the **worker** thread —
read them from the browser console (e.g. Playwright `browser_console_messages`
with `all:true`), not from `__brsConsole` (which is main-thread only).

## Known engine quirks (so you don't misdiagnose)

- The engine **auto-runs** only when `execSource:"auto-run-dev"` is passed and
  it waits for `deviceData.assets` (common.zip) to load before `execute()` —
  both already handled in `app.js`. Skipping either is what produced the
  "splash but no menu" / "-22 / default-fonts.json" crashes during bring-up.
- `MarkupGrid` bubbles LEFT at column 0; `RowList` does **not**. Before the
  grid has reported a focused item, `itemFocused` is `-1` (not `0`).
- Standalone `Button.buttonSelected` can be unreliable in this alpha SceneGraph
  build — to switch tabs, use the side drawer (BACK) rather than the top-bar
  buttons, or DOWN+arrows into content.
- Benign noise: `[sound] Error unzipping audio files`, `[core] Invalid message
  received: [object Object]`.
- Real stream playback, Cloudflare-Turnstile / WASM-PoW providers, and exact
  on-device font metrics cannot be reproduced here. Use it for layout, focus,
  navigation and BrightScript runtime errors.
